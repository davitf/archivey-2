# Documentation Generation

## Purpose

Source-derived API documentation for Archivey. MkDocs and mkdocstrings/Griffe render
public types, docstrings, dataclass-field docstrings, enum-member docstrings, and
properties so the reference stays tied to the code.

## Related specs

| Spec | Relationship |
| --- | --- |
| `archive-data-model` | Dataclasses, enums, field docs rendered in the API reference |
| `packaging-and-extras` | Docs dependency group is separate from runtime extras |
| `archive-reading` | Public reader API included in generated reference |
| `archive-writing` | Public writer API included once implemented |
| `access-mode-and-cost` | Public cost enums and receipts documented from source |

## Requirements

### Requirement: Generate the API reference from source with MkDocs and mkdocstrings

The system SHALL provide a MkDocs site (`mkdocs.yml` plus `docs/`) whose API
reference is generated by the `mkdocstrings` Python handler through Griffe reading
the `src/` package. The reference MUST derive from code and docstrings rather than
hand-maintained API pages. The site SHALL build with a single `mkdocs build`
command using a dedicated `docs` dependency group isolated from the default
lint/test environment.

#### Scenario: source-generated API reference

| Case | Expected |
| --- | --- |
| `mkdocs build` runs with the `docs` dependency group installed | Site builds and API reference documents public symbols re-exported from top-level `archivey.__all__` |

### Requirement: Render dataclass fields and enum members from their docstrings

The system SHALL surface docstrings attached to dataclass fields and enum members in
the generated reference. Dataclass fields MUST render as a "Fields" table using
`griffe-fieldz` plus a Griffe extension that titles the section accordingly. Enum
members MUST render as a Name / Value / Description table using a Griffe extension
and custom mkdocstrings template instead of the built-in enum section with a "Type"
column. `@property` accessors SHALL fold into the owning class's field table rather
than rendering as separate sections.

#### Scenario: API-object rendering matrix

| Case | Expected |
| --- | --- |
| Enum such as `ListingCost` is generated | Each member appears with name, value, and docstring, e.g. `INDEXED` and its description |
| Dataclass such as `CostReceipt` or `ArchiveMember` is generated | Each field docstring appears in the class "Fields" table |
| Property accessor is generated | Property appears with fields instead of a separate property section |

### Requirement: Documentation build is verified in CI

The system SHALL build documentation in CI with warnings treated as errors via
`mkdocs build --strict`. A broken cross-reference, removed or renamed public symbol,
or rendering warning MUST fail the build like any other check.

#### Scenario: strict-docs CI

| Case | Expected |
| --- | --- |
| Docs reference a missing symbol or unresolved cross-reference | Strict docs build fails and surfaces the drift in CI |

### Requirement: Per-format compression-library choices are documented

The documentation SHALL include `docs/library-analysis.md`, a per-format
compression-library analysis. For each codec the library reads, it MUST name the
chosen library, alternatives considered, and decision criteria: non-seekable
support, efficient seeking, corruption detection, truncation detection,
error-reporting fidelity, install/availability, and maintenance.

Each decision SHALL be documented in full within the analysis so the rationale does
not depend on external sources that may later disappear. External origins such as
`davitf/archivey-dev#214` MAY be linked for provenance but MUST NOT substitute for a
self-contained rationale.

#### Scenario: library-analysis matrix

| Case | Expected |
| --- | --- |
| Contributor reads `docs/library-analysis.md` | Each read codec (gzip, bzip2, xz/lzma, lzip, zstd, lz4, brotli, unix-compress, deflate64, ppmd) has chosen library, rejected alternatives, and rationale |
| XZ decision is documented | Native-parser rationale is recorded in full, including why stdlib `lzma.open` and `python-xz` were rejected; external link is provenance only |
