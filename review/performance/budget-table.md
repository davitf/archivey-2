## Attribution

### ZIP read-all (2.2вЂ“2.3Г—) вЂ” stream-layer overhead, not redundant decompression

cProfile over 5 runs (`~53 ms/run` profiled): `zlib.decompress` 10 ms/run вЂ”
identical work to stdlib (archivey decodes members through its own codec layer,
`zip_reader.py:839-840`, but decompresses each byte exactly once; `--track-io`
confirms bytes_decompressed == unpacked, seeks == baseline). The rest is Python
layering per 256 KiB member:

- `DecompressorStream.read` chunk loop ~7 ms/run + `bytearray.extend`
  double-buffering and the `bytes(buffer[:n])` copy ~6 ms/run
  (`decompressor_stream.py:360-376`);
- `VerifyingStream.read` wrapper ~3 ms + `zlib.crc32` ~5 ms/run (CRC itself is
  parity вЂ” `zipfile` also CRCs every member; the wrapper layer is the overhead);
- 2вЂ“3 nested `ArchiveStream` shims per member (`archive_stream.py:256`,
  `_ensure_open` Г—1920/run).

No safety feature here explains >1.3Г— beyond CRC-parity; this is H2 in
`hotspots.md` вЂ” fixable overhead (fast-path `readall`, fewer wrapper layers), not
justified-2Г— territory.

### ZIP extract-all (2.4вЂ“3.7Г—) вЂ” half streams, half safety machinery

Same stream overhead as read-all, plus per-member filesystem work
(`extraction.py`): ~5 `lstat`s per member (10 ms/run), `mkstemp`+rename atomic
write (5 ms/run), `pathlib.resolve` (6 ms/run). The safety half is exactly what
VISION's ~2Г— band is for (stdlib does none of: atomic writes, symlink-aware
destination prep, universal path checks) вЂ” but the *sum* lands above 2Г—. If the
stream-layer half (H2) is fixed, extract plausibly falls inside the safety band
without touching the safety features.

### ZIP open+list (5вЂ“8Г—) вЂ” detection + member-model construction

Profile of 200 open+list cycles on the 8-member ZIP: stdlib `_RealGetContents` is
0.115 ms of archivey's ~0.87 ms (profiled; 0.30вЂ“0.34 ms unprofiled). The rest:
format detection ~0.2 ms (magic sniff + per-open extension-map work,
`detection.py:325`, `registry.py:152`), `_to_member` ArchiveMember construction +
registration + listing-limit accounting + per-name bidi-controls warning scan
(~0.35 ms for 8 members, `zip_reader.py:535`, `base_reader.py:706`,
`listing_limits.py:35`, `naming.py:38`). Retained memory ~25 KiB per listed
reader (fine). This is the founding "list-and-hash a million small archives"
workload: 0.3 ms Г— 10вЃ¶ в‰€ 5 minutes of pure overhead vs zipfile's ~40 s. H3.

### TAR (1.4вЂ“1.8Г—) вЂ” per-member model cost on tiny absolute times

Confirms RESULTS.md's own analysis: plain-TAR absolute deltas are ~2вЂ“3 ms on 16 MiB;
the ratio is loud because tarfile `r:` is very fast. The `.tar.gz` path through
archivey's codec layer is 1.30Г— (accel off) вЂ” the compressed-stream design is not
the problem.

### VerifyingStream вЂ” within parity, not skippable

Digest work on ZIP is CRC32 (C-speed, ~9% of read-all profile) вЂ” stdlib does the
same, so the budget cost is the Python wrapper layer, not the hash. There is no
config knob to skip container-digest verification on reads where the caller
doesn't want it (`ArchiveyConfig` has none); whether one should exist is a design
question (Q4), not a perf bug вЂ” note `zipfile` can't skip either.
